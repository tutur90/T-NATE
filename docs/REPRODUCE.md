# Reproducing the paper

This guide maps every table and figure in *"NATE and T-NATE: Efficient Transformers
for IoT Task Offloading in Edge–Fog–Cloud Computing"* to the config and command that
produces it.

All commands are run from the repository root after installing the dependencies
(`pip install -r requirements.txt`). Results are written under
`logs/<dataset>/<flag>/<policy>/<timestamp>_.../`.

> **Note on exact numbers.** Reported values depend on package versions
> (see the pinned [`requirements.txt`](../requirements.txt)), hardware, and random
> seeds. Trends are reproducible; the last decimal may differ.

---

## Scenario → config mapping

The paper evaluates three topologies. Each maps to one config directory:

| Paper scenario   | `dataset` / `flag`        | Config directory                      | Nodes |
|------------------|---------------------------|---------------------------------------|-------|
| Pakistan (8N)    | `Pakistan` / `Tuple30K`   | `configs/Pakistan/Tuple30k/`          | 8     |
| Pakistan (20N)   | `Pakistan` / `Tuple100K`  | `configs/Pakistan/Tuple100k/`         | 20    |
| Synthetic (50N)  | `Synthetic` / `50N100T150D` | `configs/Synthetic/50N100T150D/`    | 50    |

> The `Tuple50k`, `Synthetic/100N100T180D`, and `Topo4MEC` configs are **not** used
> in the paper (the manuscript explicitly excludes Tuple50k). They are kept for
> completeness but are out of scope for reproduction.

Within each scenario directory, configs are grouped by family:
`Heuristics/` (Random, RoundRobin, Greedy), `DQL/` (MLP, NATE, T-MLP, T-NATE),
and `GA/` (NSGA2, NPGA).

---

## Output layout

A single training run produces, in its `logs/.../<timestamp>_.../` directory:

- `result.csv`, `log.txt` — per-epoch metrics and run log;
- `<POLICY>_convergence.png` — the convergence figures (auto-generated by the logger);
- checkpoints (`.npz` / `.pt`);
- for non-GA policies, the **qualitative** plots (task-offloading distribution, CPU
  frequency per node, latency per link, power per task) via `VisStats`.

A multi-seed run additionally writes `multi_seed_..._summary.json` with the
mean ± std used in the main results table.

## Tables

### Heuristic baselines (Table: `tab:heuristic_results`)

`Random`, `RoundRobin`, and `Greedy` per scenario. The three Greedy variants in the
paper — Greedy (cpu), Greedy (buffer), Greedy (bw) — are selected by the `obs_type`
field in the Greedy config (`cpu`, `buffer`, or `bw`); the shipped config defaults to
`cpu` (the reference heuristic).

```bash
for S in Pakistan/Tuple30k Pakistan/Tuple100k Synthetic/50N100T150D; do
  for H in Random RoundRobin Greedy; do
    python main.py configs/$S/Heuristics/$H.yaml
  done
done
```

For the Greedy (buffer) and Greedy (bw) rows, edit `obs_type` in the Greedy config
(or copy it) and re-run.

### Main comparison over 8 runs (Table: `tab:results`)

DRL/GA methods reported as mean ± std over 8 seeds (`--n_seeds 8` → seeds `0..7`).
Run each model in each scenario:

```bash
for S in Pakistan/Tuple30k Pakistan/Tuple100k Synthetic/50N100T150D; do
  for M in MLP NATE T-MLP T-NATE; do
    python main.py configs/$S/DQL/$M.yaml --n_seeds 8
  done
  python main.py configs/$S/GA/NSGA2.yaml --n_seeds 8
done
```

Each run writes a `multi_seed_..._summary.json`; the `test.mean` / `test.std`
fields are the table entries (TDR, L, E, S). NSGA-II is expected to fail to produce
feasible solutions on Synthetic (50N), as reported.

### Feature-importance ablation (Table: `tab:ablation_val`, Pakistan 8N)

The ablation varies the input features of MLP / T-MLP via the `obs_type` list in the
config (`[cpu]`, `[buffer]`, `[bw]`, `[cpu, bw, buffer]`, `[task]`,
`[cpu, bw, buffer, task]`). Edit `obs_type` in
`configs/Pakistan/Tuple30k/DQL/{MLP,T-MLP}.yaml` and run with `--n_seeds 8`.

### Selected reward weights (Table: `tab:selected_lambdas`) and λ ternary plots (Figs. `ternary_*`)

The reward weights are chosen by a grid search over the simplex
`λ0 + λ1 + λ2 = 1`. Trigger it with the special `--search "lambda=N"` form, where `N`
is the number of grid steps (paper: step 0.05 on Pakistan 8N, 0.1 on larger scenarios):

```bash
python main.py configs/Pakistan/Tuple30k/DQL/MLP.yaml   --search "lambda=20" --num_workers 8
python main.py configs/Pakistan/Tuple30k/DQL/T-MLP.yaml --search "lambda=20" --num_workers 8
```

This writes `lambda_grid_search_progress.json` and the per-scenario ternary plots
`lambda_grid_search_{val,test}.png`. To regenerate plots from saved progress without
re-running:

```bash
python main.py configs/Pakistan/Tuple30k/DQL/MLP.yaml --plot lambda
```

To produce the combined per-scenario ternary figures across the whole `logs/` tree
(as laid out in the paper):

```bash
python utils/plot_ternary_scenarios.py logs --metric 3
```

### Hyperparameters (Tables: `tab:training_hparams`, `tab:model_hparams`)

These are not produced by a command — they are the values baked into the config
files. The reported settings live directly in each
`configs/<scenario>/DQL/<model>.yaml` (`training:` and `model:` sections). The QMC
search that found them is run with, e.g.:

```bash
python main.py configs/Pakistan/Tuple100k/DQL/T-NATE.yaml \
    --search "training.lr=1e-4,5e-5,2e-5" "training.batch_size=16,32,64" \
    --sampler qmc --n_samples 32 --num_workers 4 --device cuda
```

### Inference latency (Table: `tab:inference_latency`)

```bash
python utils/benchmark_inference.py configs/Pakistan/Tuple30k/DQL/T-NATE.yaml
python utils/benchmark_inference.py configs/Pakistan/Tuple30k/DQL/T-MLP.yaml
```

(Repeat across scenarios; timings are hardware-dependent.)

## Figures

| Figure(s) | What | How |
|-----------|------|-----|
| Convergence (`*_convergence.png`) | Per-metric train/val/test curves for MLP, NATE, T-MLP, T-NATE, NSGA-II on Synthetic (50N) | Auto-generated in the run's `logs/.../` directory by a single training run of each model |
| λ ternary heatmaps | Composite score over the reward-weight simplex | `--search "lambda=N"` then `utils/plot_ternary_scenarios.py logs` (see Tables above) |
| NSGA-II Pareto front / 3-view | Pareto front from a GA checkpoint | `python utils/plot_pareto_3views.py <path/to/NSGA2_checkpoint.npz>` |
| NSGA-II crowding distance & self-adaptation σ | Evolutionary diagnostics over generations | Emitted by the NSGA-II run into its `logs/.../` directory |
| Qualitative offloading (task distribution, CPU freq, latency per link, power per task) | Per-node behaviour of each policy on P8N / S50N | Auto-generated by `VisStats` in the run directory for any non-GA policy |

## Quick smoke test

To confirm the pipeline runs end-to-end before launching full experiments:

```bash
python main.py configs/Pakistan/Tuple30k/Heuristics/Random.yaml
```

This trains nothing (heuristic) and exercises the simulator, logging, metrics, and
visualisation paths in well under a minute.
